# Deviations log — per-entry source files

This directory is the **source of truth** for the project's deviations log.
Each entry is a separate file; the reader-facing view is rendered at
site-build time into [`changes-from-source.qmd`](../../changes-from-source.qmd)
under "The full technical log".

## Why this layout

Previously every deviation was a top-of-file insertion into a single 71 KB
Markdown monolith. Two PRs landing in the same week reliably collided on
`docs/deviations-from-source.md` because both were editing the same first line
of the same file — a positional conflict that had to be hand-resolved on every
rebase. Splitting one entry per file turns the operation from a *positional*
edit into an *additive* one, which git merges cleanly.

A stitched build artifact (`docs/deviations-from-source.md`) used to live in
git as well, regenerated by a script after every entry change. That file is
gone — it is now built into the site at render time from the per-entry files
here. See [issue #384](https://github.com/lddurbin/twelve_days_to_deming/issues/384)
for the rationale: concurrent PRs were still paying a textual-conflict tax on
the stitched artifact even though their per-entry source files merged cleanly.

## Adding a new entry

1. Create `YYYY-MM-DD-short-slug.md` here. Use today's date in New Zealand
   local time (UTC+12 NZST in winter, UTC+13 NZDT in daylight-saving months);
   the project records dates in that timezone for consistency with the
   maintainer's location. The file starts with a `## YYYY-MM-DD — title`
   heading and contains the standard fields documented in `_preamble.md`:
   **What**, **Where**, **Source reference**, **Why**, **Decided in**,
   **Landed in**.
2. Commit the new entry file. Nothing else needs to change — the next site
   render pulls all per-entry files in and emits them into the page.

When a Pending entry lands, bump its "Landed in" field to the merge commit /
PR number in the per-entry file. No regeneration step.

## Ordering

The assembler (`deviations_log_assemble()` in
[`R/functions/deviations-log.R`](../../R/functions/deviations-log.R))
sorts entry files by filename descending. The `YYYY-MM-DD` prefix produces
newest-first chronological order across days; within the same date, entries
fall in alphabetical order by slug. If you care about preserving the merge
order of multiple same-day entries, pick slugs that sort in the order you
want (the assembler does not consult anything other than the filename).

## Previewing locally

To see how the rendered log looks, run `quarto preview changes-from-source.qmd`
from the repo root and navigate to that page. The R chunk on the page calls
`deviations_log_emit()` which prints the stitched markdown with entry
headings downshifted from `##` to `###` so they nest under the page's
"Full technical log" section.

If you want the raw stitched markdown (e.g. to paste into an issue or audit
the order), source the function and run it from an R session:

```r
source("R/functions/deviations-log.R")
cat(deviations_log_assemble(downshift_headings = FALSE), sep = "\n")
```
