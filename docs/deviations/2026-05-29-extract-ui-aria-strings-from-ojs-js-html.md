## 2026-05-29 — Translation pipeline extracts user-facing UI/ARIA strings from OJS, JS and raw-HTML as distinct `ui-string` / `aria-label` segment kinds

- **What** — Extended the #323 prose + #324 r-string extractor so user-facing INTERFACE strings on three new surfaces become translatable, as two new distinct segment kinds: `ui-string` (visible UI text) and `aria-label` (assistive-technology text). JS/HTML logic and element structure are never altered — only string-literal/attribute-value bytes are touched — and the identity round-trip stays byte-identical across the `.qmd`, `.R`, and a NEW `.js` corpus (129/129 files). The backtick template-literal tokenizer and a `${…}` interpolation mask were added so JS interpolations never reach a translator and round-trip exactly.
- **Whitelist (deliberately tight)** —
  - **OJS chunks** (```` ```{ojs} ````/```` ```{ojs <name>} ````): the `label:` and `placeholder:` property values of `Inputs.textarea` / `Inputs.text` / `Inputs.button` → `ui-string`. The `createDownloadButton([...], "file.txt")` filename argument is NOT extracted (it is a filename, not a label); `rows:`/numeric properties carry no string literal.
  - **JS files** (`assets/scripts/*.js`): RHS string literal of `X.textContent = "…"` and `X.placeholder = "…"` → `ui-string`; the VALUE argument of `setAttribute("aria-label"|"placeholder"|"title", <literal>)` → `aria-label`/`ui-string` (gated on the attribute-NAME argument); the first string-literal argument of the live-region announcer `announceFunnelStatus(…)` (including the common multi-line-call form, via a cross-line announce-call tracker) → `aria-label`.
  - **Raw-HTML attributes in `.qmd`** (structural tag lines that are otherwise skipped): the values of `aria-label` → `aria-label` and `title`/`alt`/`placeholder` → `ui-string`, scanned BEFORE the line is skipped as structural.
- **Deliberately EXCLUDED** — `role=`, `class=`, `id=`, `data-*=`, `aria-controls=`, `aria-live=`, `aria-atomic=`, `aria-expanded=`, `aria-pressed=`, `tabindex=`, `headers=` and similar structural/ARIA-token attributes; ARIA role/live token VALUES (`"status"`, `"polite"`, `"true"`, `"button"`); element-id templates (`` `track-title-${id}` ``); CSS class lists (`"fe-sr-live visually-hidden"`); CSS selectors and event names (`querySelector(".fe-status")`, `addEventListener("click", …)`); file paths/filenames; non-literal RHS values (a variable, a call like `formatNet(net)` / `buildCellLabel(cell)`, a concatenation — the matcher only fires when a quote/backtick sits flush after the position); equality tests (`=== "empty"`, excluded by an assignment lookbehind); and EMPTY string literals (e.g. `cell.textContent = ""` to clear a greyed-out cell). `<summary>`/`<button>` INNER label text is already captured by #323's prose path and is not double-handled here.
- **Backtick + `${…}` handling** — `.tokenize_string_literals()` now recognises the JS backtick delimiter (its `\`-escape scan already closes backtick literals correctly). `.mask_code_string()` gained a `js_interp` pattern (`\$\{[^}]*\}`) ordered BEFORE the existing glue `{…}` pattern so a `${…}` is masked as one PUA-sentinel placeholder (never leaving a stray `$` or exposing live code). The raw segment `text` keeps the exact interpolation bytes for byte-identical reinjection; the `masked` translator surface shows only opaque tokens.
- **SCOPE DECISION — SVG `<title>`/`<desc>` inner text NOT extracted (deliberate)** — `funnel-experiment.js` builds its SVG accessibility text by string concatenation: `` svg += `<title id="${titleId}">Funnel Experiment Track</title>`; `` and `` svg += `<desc id="${descId}">Track showing positions ${min} to ${max}. ${funnelLabel}</desc>`; ``. The translatable words sit INSIDE a larger template literal that also contains SVG markup, an interpolated `id`, and (for `<desc>`) further interpolations — there is no clean call/assignment POSITION to whitelist, and the only robust handle would be parsing the SVG markup inside the literal, which is exactly the kind of structural rewriting this pipeline avoids. Extracting the whole literal would put `<title id="…">` markup on the translator surface and risk corrupting the SVG. These two strings ("Funnel Experiment Track" and "Track showing positions … . …") are therefore scoped OUT of this pass and remain English. They are decorative/illustrative accessibility text on an interactive funnel widget; a follow-up can add them via a targeted rule or by refactoring the JS to hoist the human text into a `setAttribute`/`textContent` position the existing whitelist already covers. (Two further JS strings — the ternary-branch consts `"No stage active."` / `"Funnel always at"`/etc. and the `themeState.textContent = on ? "Dark" : "Light"` toggle states — are likewise excluded: they live at ternary positions, not at a whitelistable assignment/call position. Same rationale; same follow-up path.)
- **Where** — Tooling only: `R/translation/code-string-extract.R` (backtick tokenizing; `${…}` mask; OJS/JS/HTML whitelist tables + matcher `.ojs_js_literal_kind()`; `.scan_ojs_js_line_literals()`; `.scan_html_attr_values()`/`.one_literal_at()`; cross-line `.update_announce_call_context()`), `R/translation/prose-extract.R` (OJS chunk detection + dispatch; raw-HTML attribute scan before the structural skip; `extract_js_file()`; `js_corpus()`; optional `context` hint on sub-line segments), `R/translation/roundtrip.R` (gate now covers the `.js` corpus). New test file `tests/testthat/test-ui-string-extract.R`. No course content, no rendered output, and no `assets/scripts/*.js` source bytes change.
- **Source reference** — n/a. Translation-infrastructure work; nothing in Neave's source materials is affected. The strings extracted are the same input labels, button captions, and accessibility text the interactive widgets already display.
- **Why** — Issue #325: the French edition must be able to translate the interface (input labels, button text, screen-reader announcements, ARIA labels), not just prose and chart labels. Those strings live in OJS chunks, JS assets, and raw-HTML attributes. The whitelist is conservative by design — a false positive that rewrote an element id, a CSS class, an ARIA role token, or a selector would corrupt the UI, which is far worse than a missed label (addable in a follow-up). The new kinds plus the optional `context` hint give a translator the interface context required by the issue's AC #1.
